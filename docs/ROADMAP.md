# Дорожная карта

Порядок не случаен: сначала ядро и доменная модель, потом UI, в конце — генерация и LLM.

## Основной трек (редактор)

| Фаза | Что делаем | «Готово» = |
|---|---|---|
| **0. Скелет** | Монорепо, `core` каркас, канвас: pan/zoom, **сетка + привязка**, координаты в мм. Command-стек (undo/redo) | Вид двигается, точки липнут к сетке, undo работает |
| **0.5 Оболочка + релиз-пайплайн** | **Выбор Tauri vs Electron** (UI за `Platform`-портом → обратимо). GitHub Actions: сборка/подпись/публикация + манифест авто-обновления. **Публичный ключ updater + версия схемы `.seeproj` — с первого релиза** | `.exe` ставится и сам обновляется |
| **1. Лист + штамп** | Формат А3/А4, рамка (ГОСТ 2.301), зонная сетка (1–8 / A–F), основная надпись (ГОСТ 2.104, шаблонная) | Печатается корректный лист со штампом |
| **2. Символы + библиотека** | Формат `*.symbol.json`, рендер УГО, расстановка/поворот/зеркало, snap выводов. 10–12 базовых УГО (QF, УЗО, QS, KM, K, XT, SB, HL, M, T, FU) | Элементы расставляются с позобозначениями |
| **3. Провода + связность** | Провода (≠ линии), junction'ы, **union-find**, **3-Wire/1-Wire**, разрез провода при вставке символа, contact cross (master/slave по имени), **автонумерация** (ГОСТ 2.709), auto-reconnect | Провода нумеруются по цепям, контакты ссылаются на катушки |
| **4. Каталог + генерация** | `*.part.json`, привязка `Device`→`CatalogPart` (неск. кодов), master/slave + кросс-рефы. Отчёты: перечень элементов, перечень/план клемм, кабели/жилы, документы. Экспорт PDF/Excel | Из схемы — спецификация, таблица соединений, перечень клемм |
| **5. Редактор блоков** | Создание своих УГО + выводы → `*.symbol.json` | Можно нарисовать и сохранить свой символ |
| **6. Клеммники + разъёмы** | `Terminal`/`Connector` (number/sorting/pins), клеммник XT, варианты генерации (табличная/графическая/многоуровневая), Wire2Terminal | Клеммник и разъёмы с раскладкой и таблицей |
| **7. Шкафы (забудова)** | Модуль компоновки (масштаб 1:10): шкаф/рейки/кабель-каналы, «список символов для вставки» (связь схема→шкаф через `Device`), габариты из каталога, размеры | Физическая компоновка шкафа, связанная со схемой |

## Трек: LLM-генерация (после MVP)

1. **Core API** — низкоуровневые команды (`addComponent`, `connect`, `numberWires`, `generateBOM`, `renderSheet`).
2. **Skills/tools** — высокоуровневые операции (`addBreaker("QF1","16A","C")`, `makeBusbar`, `dropToTerminals`).
   Ориентир — лента See Electrical (см. `REFERENCE-SEE-ELECTRICAL.md`): каждая группа команд = кандидат в skill.
3. **Declarative spec / DSL** — LLM описывает щит декларативно, компилятор раскладывает/соединяет/нумерует.
   Прецедент: в See Electrical есть **auto-diagramming** — генерация схемы прямо из Excel-таблицы. Наш DSL/спека — тот же принцип, но для LLM.
4. **Авто-раскладка** — для типовой топологии щита (рейки, вертикальные спуски).

## Трек: Open-source / сообщество

- JSON Schema для `*.symbol.json` / `*.part.json` + валидация в CI.
- Библиотечные паки (symbols + parts + manifest + версия + лицензия).
- Community-реестр для публикации/установки паков.
- Решить лицензии: код (MIT/Apache-2.0?) vs контент (CC-BY?).

## Трек: облако / мультиюзер (опционально, если проект «взлетит»)

Сейчас НЕ делаем, но `core` держим готовым (принцип 7 / раздел «Дистрибуция» в ARCHITECTURE):
1. **Web/PWA** — тот же UI, один пользователь.
2. **Backend-сервис** — `core` на Node, проекты в БД/объектном хранилище, авторизация + тенанты во внешнем слое.
3. **Realtime-совместная работа** — синхронизация лога команд (CRDT/OT/серверный порядок).

Модель — **open-core**: ядро/десктоп открыты, хостинг — платный.
