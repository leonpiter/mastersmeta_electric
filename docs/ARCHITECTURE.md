# Архитектура

## Слои

```
┌─────────────────────────────────────────────┐
│  UI / Canvas (рендер, ввод, инструменты)      │  apps/desktop — зависит от стека
├─────────────────────────────────────────────┤
│  Commands (undo/redo, command-паттерн)        │
├─────────────────────────────────────────────┤
│  Domain (модель, движок связности)            │  packages/core — ЯДРО, без UI
├─────────────────────────────────────────────┤
│  Library (УГО) · Catalog · Generators         │
├─────────────────────────────────────────────┤
│  Persistence (формат файла проекта)           │
└─────────────────────────────────────────────┘
```

`core` работает headless (Node) и в браузере. GUI и будущий LLM-агент — два клиента одного API.

## Доменная модель — три слоя

Ключевой вывод из разбора See Electrical: **каталог изделий, графические УГО и схема — разные сущности**,
а одно устройство имеет несколько графических представлений (master/slave).

### 1. Каталог (Components — изделия производителей)

```ts
Manufacturer { id, name }                       // ABB, Schneider, Legrand...
CatalogPart {
  id, manufacturerId, category,                 // "Auxiliary Contactors"
  partNumber,                                    // "201080"
  description, componentCode,                    // "K" (ГОСТ 2.710)
  functionalUnits: FunctionalUnit[],             // катушка + контакты
  cabinet: { width, height, depth, symbolId },   // габариты X/Y/Z + физ. символ для листа шкафа
  props: { цена, ... }                           // для спецификации
}
FunctionalUnit { name, defaultSymbolId, pins: Pin[] }  // "A1,A2" / "13,14" ...
```

### 2. Символы (Symbols — графические УГО)

```ts
Symbol {
  id, name, category,
  componentCode,                 // префикс позобозначения (K/S/Q/X/...)
  kind,                          // поведение (See): coil | component | component-aux | contact-NO | contact-NC | terminal | connector | black-box
  graphics: GraphicPrimitive[],  // линии/дуги/окружности/текст
  pins: Pin[]                    // { name: "1"|"A1"|"13", x, y }
}
```
Декуплено от каталога: `FunctionalUnit` ссылается на `Symbol`. Сообщество может вкладывать УГО и каталог независимо.

### 3. Схема (чертёж)

```ts
Project {
  meta, library, catalog,
  documents: Document[]                 // типизированные модули (см. ниже)
}
Document { type, name, pages: Page[] }  // cover | circuit | distribution | cabinet | lists | ...
Page { format(А3/А4), titleBlock, zoneGrid, content }

Device {                                 // ЛОГИЧЕСКОЕ устройство = 1 строка спецификации
  id, designation: "KM1", componentCode,
  catalogPartRefs: string[],             // НЕСКОЛЬКО кодов (в See разделены `;`): база + доп. контакты
  props
}
// Kod katalogowy задаёт конфигурацию контактов устройства → ею наполняется contact cross.
SymbolInstance {                         // ГРАФИЧЕСКОЕ размещение (master/slave: N на устройство)
  id, deviceId, functionalUnitRef, symbolId,
  pageId, x, y, rotation, mirror, showLabels
}
Terminal extends Device {                // клемма — спец. сущность
  terminalNumber, terminalSorting, stripId   // принадлежит клеммнику XT; уровни (multilevel/3D)
}
Connector extends Device {               // разъём — как клемма: Sigla(имя) + pinName + pinId
  pins: { pinId, name }[]                // номера/имена пинов могут браться из каталога
}
Wire { points[], endpoints, type, locked, number } // ЭЛЕКТР. связь (≠ Line!); type: power|control; locked — не перенумеровывать; нумеруется только между 2 символами
Line { points[] }                        // просто графика, НЕ электрическое соединение
Cable { product, cores: CableCore[] }    // группирует провода, которые пересекает (жилы: No./Colour/Size)
Junction { x, y }                        // узел: соединение 3+ проводов
CabinetItem {                            // ФИЗИЧЕСКОЕ представление Device в шкафу (вид «забудова»)
  deviceId, x, y, width, height,         // габариты берутся из каталога (Kod katalogowy)
  railId?                                // привязка к монтажной рейке (двигается вместе с ней)
}
```

`Net` / `Potential` и `CrossReference` — **вычисляются**, не хранятся (см. ниже).

## Движок связности (сердце)

1. Собрать точки соединения: концы проводов, выводы (`pin`) `SymbolInstance`, junction'ы.
2. Совпадающие по координатам (после привязки к сетке) объединить через **union-find**.
3. Каждое множество = одна **цепь (Net) = один потенциал**.
4. **Автонумерация по ГОСТ 2.709**: цепи присваивается номер (MVP — сквозной, с ручным override).
5. Из этой же структуры — **таблица соединений** (`QF1:2 → KM1:A1, провод №5`).
6. Провода привязаны к выводам → при движении `SymbolInstance` концы едут за ним (auto-connection/rubberband).
7. **Сквозная связность (S29):** локальные цепи листов считаются по п.1–3, затем
   `computeProjectNets(pages)` объединяет цепи разных листов, на которых стоят **соединители
   страниц** (`SymbolKind "page-connector"`) с одинаковой меткой `SymbolInstance.signal` — так
   потенциал «переносится» между листами одной цепью. Таблица соединений строится по этим
   сквозным цепям; служебные выводы соединителей в неё не попадают.

## Кросс-референсы

- **Соединители страниц (S29):** адресная стрелка (`page-connector`, 1 вывод) на конце провода;
  соединители с общей меткой `signal` электрически связаны (см. движок связности) и адресуют
  друг друга `лист·зона` (через `zoneOf` + графа 7 штампа). Одиночный клик по адресу — переход
  на лист партнёра. Метка по умолчанию = номер цепи (`circuitNumberAt`).

- **Связывание master/slave по общему имени:** контакт получает то же `Product (-)`, что и катушка (напр. `K1`) +
  номера выводов (IEC: катушка `A1/A2`, NO `13/14`, NC `21/22`). У катушки авто-рисуется **«contact cross»**
  (зеркало контактов) со ссылками на листы/зоны её контактов. «Список контактов» ловит несвязанные.
- Между потенциалами на разных листах (нотация `→3.1` = лист 3, цепь 1), по **зонной сетке** (колонки 1–8, строки A–F).
  Копирование потенциалов на следующий лист создаёт кросс-рефы автоматически.

## Форматы файлов (открытые)

- `*.symbol.json` — УГО (графика + pins + метаданные). JSON Schema для валидации.
- `*.part.json` — каталожное изделие (functionalUnits + pins + артикул).
- **Библиотечный пак** — папка/zip: `symbols/` + `parts/` + `manifest.json` (имя, версия, автор, лицензия). Публикуется в community-реестр.
- **Проект** `*.esch` — zip-контейнер: `project.json` + встроенные копии используемых символов/партов (самодостаточность).

## Механика (уроки из SEE Electrical V8R2)

### Wire ≠ Line (инвариант)
Провод (`Wire`) — реальная электрическая связь; линия (`Line`) — просто графика; только провода участвуют в связности.
Вставка символа на провод **автоматически разрезает** его и подключает; на линии — нет (типичная ошибка пользователя).

### Авто-подключение
- **3 Wires** (Ctrl+3): рисуешь одну связь L1→цель — параллельно авто-рисуются L2→/L3→ (3 фазы).
- **1 Wire** (Ctrl+1) — одиночный. Провода **режутся в точках, где стоят компоненты**; узлы на потенциалах появляются сами.

### Нумерация и шаблоны
- Политики имён: **сквозная** или **по номеру листа** (`1CB1` = лист 1 + CB + 1).
- **Page Index** (`1a`) — вставка листа без перенумерации существующих; кросс-рефы обновляются.
- Шаблон workspace задаёт: стандарт (IEC/IEEE) · схему нумерации · формат листа · число зон.

### Block → Component
Выделил графику → `Block` → выбрал «Component» + префикс имени = свой компонент. `Explode` — для правки. Основа редактора блоков.

### Сетка
Выводы символов — на шаге **5 мм**; графику можно рисовать в шаге 1 мм при «опорной сетке» 5 мм. Начало координат (0,0) — левый нижний угол.

## Отчёты (генераторы)

Авто-генерируются из связности и каталога; шаблоны — `.tdw` + Crystal Reports; экспорт в Excel.
- **Перечень элементов** (Zestawienie aparatury) — спецификация.
- **Перечень/план клемм** (Zestawienie zacisków) + графический план клеммника (Matrix).
- **Перечень кабелей** и **жил кабелей** (Zestawienie kabli / żył).
- **Каналы PLC**, **перечень документов**.

Два режима: «живые» списки (модуль *Технич. база проекта*) и генерируемые листы (модуль *Zestawienia*, right-click → Generuj).

## Уроки из V7-мануала (модель)

- **`Symbol.kind`** (выбор при Crea blocco): `coil` (авто contact cross), `component` (уник. сигла, напр. двигатель), `component-aux` (уник. + доп. контакты, напр. выключатель), `contact-NO/NC`, `terminal`, `connector`, `black-box`.
- **Провода:** `type` (power/**control**), `locked` (авто-нумерация не трогает), номер только если провод между двумя символами.
- **Лист (шаблон):** число колонок (=зоны), позиции верх/низ потенциалов, отступы, расстояние до contact mirror, **верхний отступ авто-связи 25 мм**, шаг сетки ~4 мм. Атрибуты: **проектные** (раз на проект) vs **листовые** (на каждый лист).
- **Каталог → шкаф:** в коде габариты X/Y/Z + «Simbolo Quadro»; связь схема↔шкаф через артикул.
- **Слои:** до 512 (скрытие на печати).
- **Отчёты:** генерируются как листы; индекс проекта из атрибута листа *Titolo*; **Excel round-trip** (DBListsToExcel/FromExcel, коды через `;`).

## ГОСТ (ЕСКД)

| Стандарт | О чём |
|---|---|
| ГОСТ 2.301 | Форматы (А4 210×297, А3 297×420), рамка |
| ГОСТ 2.104 | Основная надпись (штамп), формы 1/2/2а |
| ГОСТ 2.710 | Буквенные коды: QF, QS, KM, KK, KT, SB, XT, FU, HL, M, T |

## Дистрибуция и поставка

**Сейчас (по умолчанию):** десктоп-инсталлятор, 100% офлайн, проекты — файлы `.esch`. Обмен библиотеками — файлы/GitHub/статика (без сервера).

**Оболочка — решение отложено до Фазы 0.5 (Tauri vs Electron), и оно обратимо:** UI — обычный web (Vite + SVG), вся логика — в `core`; нативные вызовы (диалоги файлов, обновление) — за тонким `Platform`-портом. Смена оболочки = переписать адаптер (дни), не приложение.

| | Tauri | Electron |
|---|---|---|
| Размер | ~3–10 МБ | ~80–150 МБ |
| RAM | ниже (системный webview) | выше (свой Chromium) |
| Рендер | WebView2/WKWebView/WebKitGTK (Linux — слабое звено) | одинаковый Chromium везде |
| Бэкенд / экосистема | Rust (нужен toolchain) | Node/JS, огромная экосистема |
| Обновление | встроенный updater | electron-updater (зрелый) |

Для CAD-инструмента размер не критичен → **Electron безопаснее на долгий срок** (консистентность рендера, экосистема, чистый JS — удобнее для агентной разработки); **Tauri** выигрывает в размере/RAM/встроенном updater. Решаем на Фазе 0.5.

### Авто-обновление (заложить с ПЕРВОГО релиза)

Механизм один для обеих оболочек (Tauri — встроенный updater; Electron — `electron-updater`). Поток:
1. При старте (если есть интернет) приложение запрашивает манифест (`latest.json`) на **GitHub Releases** (без сервера).
2. Сравнивает версию (semver) → если новее, **уведомляет**: «Доступна версия X.Y — Обновить / Позже».
3. Скачивает артефакт, **проверяет подпись**, ставит, перезапускается.

Две подписи (разные!):
- **Подпись обновления** — *обязательна*; **публичный ключ зашит в приложение** → заложить с **первого** релиза, иначе ранние установки не смогут авто-обновляться (от выбора оболочки не зависит).
- **Подпись кода ОС** (Authenticode на Windows) — против предупреждений SmartScreen.

Релиз: git-тег → **GitHub Actions** собирает/подписывает/публикует артефакты и манифест. Инфраструктура — ноль стоимости.

Поведение: офлайн → молча пропускает (приложение полностью работает); пользователь может отложить/отключить. Updater — в `apps/desktop` (оболочке), не в `core`.

**Связь с миграциями:** авто-апдейт → пользователи прыгают через версии → **версия схемы `.esch` + миграции** (принцип 7) обязательны: новая версия открывает старый файл и мигрирует его.

**Спектр на будущее (один и тот же `core`, без переписывания логики):**
1. **Десктоп** (сейчас) — локальные файлы.
2. **Web/PWA** — тот же UI в браузере, один пользователь, local storage / File System Access API.
3. **Backend-сервис** — `core` на Node, проекты в БД/объектном хранилище, REST/WebSocket; авторизация и тенанты — во внешнем слое. Headless-`core` на сервере: LLM-генерация, валидация, рендер SVG/PDF.
4. **Realtime-совместная работа** — лог команд синхронизируется по WebSocket; разрешение конфликтов (CRDT/OT/серверный порядок). Команды уже first-class (undo/redo) → это расширение, не переписывание.

**Что закладываем в `core` сейчас, чтобы переход был бесшовным** (принцип 7 в CLAUDE.md):
- ноль прямого I/O — хранилище через инжектируемый порт `Storage`;
- стабильные `uuid` у всех сущностей (не индексы массивов);
- детерминированная сериализация проекта + версия схемы и миграции;
- никаких глобальных синглтонов — много проектов/сессий в одном процессе (мультитенант = много инстансов `core`);
- сериализуемые команды (undo/redo = будущий op-log совместной работы);
- авторизация/пользователи/тенанты — ВНЕ ядра.

**Когда оправдан VPS + мультитенант:** только если решишь делать платный облачный сервис или онлайн-совместную работу. Модель — **open-core**: код ядра/десктопа открыт (MIT/Apache), хостинг — платный продукт. Это бизнес-решение, не требуется для самого инструмента.
