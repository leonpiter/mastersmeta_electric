# S23 · AI-first режим: со-проектировщик + база инженерных знаний

- **Веха:** F — LLM-генерация
- **Статус:** 📝 Запланирован
- **Зависит от:** S14 (Core API), S15 (Skills), S16 (DSL/авто-раскладка). **Блок A (база знаний) можно вести рано** — параллельно редактору, не дожидаясь MVP (как S12).
- **Ветка:** — (плановый документ; код — после MVP. Ветку завести при старте конкретного блока.)
- **Оценка:** крупный, дробится на блоки A–D (каждый ~неделя+); см. секвенс.

## Цель
Режим, где инженер **словами описывает свой щит** (как работает, какие задачи решает), а агент —
**задаёт уточняющие вопросы, согласует логику работы** и собирает корректную схему через типизированный
API/skills, а ядро её детерминированно валидирует. Качество проектирования обеспечивает **открытая база
инженерных знаний** (силовые / слаботочные щиты), подаваемая модели через системный промпт + RAG —
**без дообучения модели**.

## Контекст
Существующая Веха F (S14–S16) — «механический движок»: *как программно собрать схему* (Core API →
skills → DSL/авто-раскладка). В ней нет трёх слоёв из видения AI-first:
1. **База знаний** — чтобы агент проектировал грамотно и задавал *правильные* уточняющие вопросы по типу щита.
2. **Диалоговый режим-интервью** — со-проектировщик, а не «одна кнопка генерации».
3. **LLM-клиент** за портом — собственно подключение модели.

Этот спринт достраивает их *поверх* S14–S16. Решения согласованы с пользователем (AskUserQuestion,
2026-05-26) и зафиксированы в `docs/AI-FIRST.md` (создаётся блоком A):

- **Подход к знаниям:** курируемая открытая база + **RAG** + skills. **НЕ дообучать модель** — домен
  нишевый, ГОСТ/нормативный, меняется, нужна проверяемость и ссылка на источник; дообучение «зашивает»
  знания непрозрачно, дорого итерировать, плохо ложится на open-source-вклад. (Fine-tune — возможная
  поздняя оптимизация, не старт.)
- **Где знания:** человекочитаемые методички `docs/knowledge/*.md` + машиночитаемые правила `*.rule.json`
  (открытый версионируемый формат, как `*.symbol.json` / `*.part.json`; шарятся как библиотечные паки).
- **Принцип «агент предлагает — ядро проверяет»** (принципы CLAUDE.md №1 API-first, №6 тесты = «глаза»,
  №7 ключи/IO вне ядра): LLM вызывает skills → `core` исполняет и валидирует (union-find связность,
  автонумерация ГОСТ 2.709, спецификация, `danglingPins`) → headless-рендер SVG даёт агенту «увидеть»
  результат и самокорректироваться.

## Архитектура слоёв (целевая)
```
AI-first UI: чат-панель (интервью, уточнения, превью)        apps/desktop
  ↓
LLM-клиент (provider-agnostic, BYO key, tool-use loop)       порт LlmClient
  • интерфейс LlmClient — в packages/agent (reusable на сервере)
  • провайдер + инжекция ключа — в apps/desktop (ключ вне core)
  ↓
Оркестратор-агент (интервью → спека → сборка → само-проверка) packages/agent
  ▲ читает Базу знаний (системный промпт-плейбуки + RAG)
  ↓
Skills (S15) → Core API (S14) + headless-рендер SVG (S22/B)  packages/core
  ↓
Домен + движок связности (валидация)                         packages/core
```

## Scope (задачи)

### Блок A — База инженерных знаний *(можно начинать рано; ядро ответа «куда складывать инструкции»)*
- [ ] `docs/AI-FIRST.md` — документ-стратегия: видение, архитектура слоёв, подход к знаниям (почему RAG, не fine-tune), диалоговый цикл, LLM-интеграция, авто-визуализация, дорожная карта, открытые вопросы.
- [ ] `docs/knowledge/` + `README.md` — конвенции каталога: методички (`*.md`) и правила (`*.rule.json`).
- [ ] Формат `*.rule.json` + JSON Schema (по образцу `packages/core/schemas/symbol.schema.json`): поля `id`, `panelType` (power/low-current), `applicability` (условие), `recommendation`, `source` (ГОСТ/ПУЭ/норматив), `topology?`, метаданные пака (`manifest` + версия + лицензия).
- [ ] Seed-контент (минимальный, для проверки формата): 1 плейбук силового щита + 1 слаботочного (методичка + 2–3 правила каждый) + список уточняющих вопросов по типу щита.
- [ ] Таксономия: силовые (ВРУ, распределение, УЗО-группы, силовые спуски, селективность) · слаботочные (СКС, PoE, заземление/экранирование) · электробезопасность (ПУЭ, расстояния) · ГОСТ-оформление · типовые топологии.

### Блок B — LLM-порт (provider-agnostic)
- [ ] Интерфейс `LlmClient` в `packages/agent` (по образцу порта `Storage`/`Platform`): `complete()` + tool-use loop; ноль секретов/IO в `core`.
- [ ] Конкретный провайдер (по умолчанию Claude) + ввод/хранение BYO-ключа — в `apps/desktop` (через `Platform`).
- [ ] Загрузка/индекс базы знаний для RAG: на старте — keyword/тег-поиск по `*.rule.json` (эмбеддинги — открытый вопрос).

### Блок C — Оркестратор + диалоговый цикл
- [ ] `packages/agent`: цикл `интервью (вопросы по плейбуку) → черновая спека (DSL S16) → согласование → сборка через skills (S15) → авто-валидация ядром → headless-рендер → само-проверка агента → превью → правки → итерация`.
- [ ] Системный промпт собирается из методичек + релевантных правил (RAG) под тип щита.
- [ ] Чат-панель в `apps/desktop` (интервью, показ уточняющих вопросов, превью SVG, кнопки «принять/править»).

### Блок D — Авто-визуализация щита *(задел; расширяет S10)*
- [ ] Расширить `CatalogPart` ссылкой на визуальную модель оборудования (2D-footprint / 3D-ref).
- [ ] Агент расставляет оборудование на DIN-рейки (поверх модуля «забудова» S10), подбирает оболочку.
- [ ] Пользователь правит раскладку.

## Технический дизайн
- `core` не меняется по существу (домен): агент — внешний клиент через S14/S15. Само-верификация — на `renderSheetToSvg`/связности.
- `LlmClient` и оркестратор — в `packages/agent` (headless, reusable на сервере; ARCHITECTURE «Дистрибуция» п.3). Ключ инжектируется на границе оболочки/сервера, **не в ядре** (принцип №7).
- База знаний — открытый контент, версионируется в git и пакуется как библиотечный пак (manifest + версия + лицензия), как УГО/каталоги.
- RAG на старте — простой (теги/keyword по `panelType`+условию); переход на эмбеддинги — при росте корпуса.

## Открытые вопросы
- Провайдер по умолчанию и модель; формат BYO-ключа (где хранит оболочка).
- RAG: keyword/теги vs эмбеддинги — порог перехода; где индекс (в паке или строится локально).
- Лицензия паков знаний — это *контент* (как УГО/каталоги): предположительно CC-BY (уже открытый вопрос к S12 в CLAUDE.md) — синхронизировать.
- Как валидировать «инженерную корректность» вывода агента сверх формальной связности (нормативные проверки?).
