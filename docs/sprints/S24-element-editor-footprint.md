# S24 · Редактор элемента (УГО + footprint 1:1) → конструкторская сборка шкафа

- **Веха:** C — Создание контента (расширяет S9 «Редактор символов» и S10 «Шкафы»)
- **Статус:** 📝 Запланирован
- **Зависит от:** S6 (каталог), S9 (редактор символов), S10 (компоновка шкафа). **После MVP** (S1–S7).
- **Оценка:** крупный, дробится на блоки A–D.

## Цель
Замкнуть путь **«принципиальная схема → конструкторский чертёж щита»** в одной программе:
у каждого УГО на схеме можно выбрать реальный прибор, у прибора есть **физический footprint в масштабе 1:1**,
а в разделе «Шкафы» из этих приборов собирается конструкторская компоновка (Visio/CAD-канвас) и подбирается оболочка.

## Решения (согласовано с пользователем, 2026-05-26)
- **Где визуализация:** 1:1-footprint прибора живёт **в каталоге** — `CatalogPart` получает графику footprint (а не отдельная «библиотека визуализации»). Один открытый формат `*.part.json`. Цепочка: `SymbolInstance` (УГО) → `Device` → `CatalogPart` → footprint.
- **Редактор:** **один редактор элемента с двумя вкладками** — «Схемное УГО» и «Физический footprint (1:1)»; общий движок рисования примитивов (из S9). Оба представления привязаны к одному типу/устройству.
- **Приоритет:** после MVP (сначала критпуть S3→S7).

## Сценарий (end-to-end)
1. Инженер чертит **принципиальную схему** (УГО: реле, клеммы, контакторы, лампы) — S2/S3/S5.
2. Каждому УГО на схеме **выбирает реальный прибор** из каталога — S6 (у прибора есть footprint 1:1).
3. В разделе **«Шкафы»** — CAD-канвас (как Visio): расставляет **footprint'ы** выбранных приборов на DIN-рейки/в оболочку (масштаб 1:1, привязка, размеры), **подбирает оболочку** по габаритам/модульности — S10.
4. Результат — конструкторская сборка щита, связанная со схемой через `Device`.

## Модель данных
- `CatalogPart.footprint`: `{ graphics: GraphicPrimitive[]; pins?: Pin[]; width; height; depth }` (мм, 1:1). Габариты X/Y/Z уже предусмотрены в ARCHITECTURE («Каталог → шкаф»).
- `SymbolDef` (УГО) и `footprint` используют **общий** `GraphicPrimitive` (один движок рисования) — отличаются масштабом/назначением.
- `CabinetItem` (физическое размещение в шкафу) рисуется из `Device.catalogPart.footprint` — связь схема↔шкаф через `Device` (уже в модели).

## Scope (блоки)
### Блок A — Редактор элемента (расширяет S9)
- [ ] Единый редактор: вкладки «УГО» и «Footprint (1:1)»; общий примитив-редактор (линия/прямоуг./дуга/окруж./текст + выводы на сетке 5 мм).
- [ ] Сохранение: УГО → `*.symbol.json`; footprint → в `*.part.json` (поле `footprint`).

### Блок B — Footprint в каталоге (расширяет S6)
- [ ] `CatalogPart.footprint` + JSON Schema; габариты X/Y/Z; точки крепления/клеммы.
- [ ] «Библиотека визуализации» = footprint-сторона каталога (один формат `*.part.json`, паки как у УГО).

### Блок C — Привязка на схеме (использует S5/S6)
- [ ] У `SymbolInstance`/`Device` — выбор реального прибора (каталог) → подтягивается footprint и контакты.

### Блок D — Конструкторская сборка шкафа (расширяет S10)
- [ ] CAD-канвас (1:1): «список приборов для вставки» (из `Device`'ов схемы), расстановка footprint'ов на рейки/каналы, привязка, размеры.
- [ ] Подбор оболочки (по габаритам/числу модулей); индекс-таблица материалов.

## DoD
- [ ] В редакторе элемента у типа задаётся и УГО, и 1:1-footprint; оба сохраняются в открытые форматы.
- [ ] На схеме УГО привязывается к реальному прибору из каталога (с footprint).
- [ ] В «Шкафах» footprint'ы выбранных приборов расставляются 1:1 и подбирается оболочка; связь со схемой через `Device`.
- [ ] `pnpm verify` зелёный; headless-рендер (S22/B) переиспользуется для footprint.
