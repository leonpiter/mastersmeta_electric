# Создание нового спринта (проект see_elecctrical_copy)

Создаёт корректно спланированный спринт-документ. Не «механическое» создание файла — это постановка ТЗ.

Конвенции проекта: solo-разработка (агент — Claude Code), монорепо TS, спринты в `docs/sprints/S{N}-{slug}.md`, индекс `docs/SPRINTS.md`, дорожная карта `docs/ROADMAP.md`, ветки от `main`. Никакой шины/`/dispatch`/forgejo/BACKLOG — это не используется.

## Шаги

### 1. Согласование (если зона риска)
Через AskUserQuestion (1 вопрос): «Спринт затрагивает доменную модель `core` / движок связности / формат файла `.seeproj`?»
- **Да** → задай 2–3 уточнения (одним опросом): что меняется по сути; какие смежные части затрагиваются (символы/нумерация/отчёты/миграции схемы); регрессионные риски. Ответы → в раздел «Контекст».
- **Нет (UI / документация / точечный фикс)** → пропусти, к шагу 2.

### 2. Контекст (не placeholder!)
Через AskUserQuestion спроси 2–3 предложения: «Зачем спринт, какую проблему решает?». НЕ оставляй заглушку шаблона.

### 3. Номер спринта
Glob `docs/sprints/S*.md`, извлеки номера regex `S(\d+)-`, найди максимум, следующий = max+1. Формат — `S{NN}` (две цифры, напр. `S20`). **Не используй `grep -oP`** (ломается на Windows MinGW).

### 4. Slug
Из `$ARGUMENTS` (kebab-case, латиница/цифры/дефис) или спроси через AskUserQuestion (slug + человекочитаемое название).

### 5. Разведка дубликатов
Grep slug-словами по `docs/sprints/S*.md` (заголовки/цели). Если похожее есть — предупреди и спроси «новый / объединить».

### 6. Метаданные
Через AskUserQuestion: **Веха** (A–G из ROADMAP) и **Оценка** (напр. «1–2 дня», дефолт «1–3 дня»). Статус = `📝 Запланирован`. Ответственный — Claude Code (solo, не спрашивать).

### 7. Создать файл по шаблону
Прочитай `docs/sprints/SPRINT-TEMPLATE.md`. Создай `docs/sprints/S{NN}-{slug}.md`:
- шапка: Веха, Статус `📝 Запланирован`, Зависит от (спроси/выведи), Ветка (см. шаг 9), Оценка;
- **Цель** + **Контекст** — из шагов 1–2 (реальный текст);
- **Scope**, **Технический дизайн**, **Acceptance / DoD** — пункты `- [ ]` (наполни по сути спринта; не оставляй пусто);
- **Итоговый отчёт** — заглушка (заполнит `/sprint-finalize`).

### 8. Синк с ROADMAP / индексом
- `docs/ROADMAP.md`: через AskUserQuestion спроси, к какой вехе отнести; добавь строку через Edit.
- `docs/SPRINTS.md`: добавь ссылку на новый спринт в индекс-таблицу.

### 9. Предложить ветку
Через AskUserQuestion: создать `sprint-{N}-{slug}` (или `phase-…`) **от `main`**? Если да:
```bash
git checkout main && git pull --ff-only 2>/dev/null; git checkout -b <ветка>
```
(`git pull` пропусти, если нет remote.)

### 10. Финал
```
✅ Создан S{N}: <название>
📄 docs/sprints/S{NN}-{slug}.md
🌿 Ветка: <ветка / нет>
🗺 ROADMAP: веха «<…>» · индекс SPRINTS.md обновлён
Следующий шаг: уточнить Scope/Acceptance и приступать (или /sprint {N} для статуса).
```

## Anti-patterns
1. Не оставляй контекст/цель заглушкой — спрашивай у пользователя.
2. Не используй `grep -oP` — только Glob + regex в коде.
3. Не выдумывай несуществующую инфраструктуру (шина, /dispatch, forgejo, BACKLOG) — её в проекте нет.
4. Не пропускай согласование для зоны модели/связности/формата файла.

## Быстрый режим
Если сказано «без вопросов» / `--quick` — только шаги 3, 4, 7, 9 (контекст останется кратким, ROADMAP/индекс не трогаем).
