# Проект: open-source редактор электрических схем (See Electrical lite)

> Инструкции для агентов (Claude Code). Читать перед работой.

## Что это

Офлайн десктоп-редактор принципиальных электрических схем щитов — упрощённая,
ГОСТ-ориентированная альтернатива See Electrical / AutoCAD Electrical.
Разработка **полностью агентная** (код пишет Claude Code).

**Дальние цели:**
- **Open-source с сообществом** — инженеры создают и шарят свои УГО и каталоги компонентов.
- **LLM-генерация схем** — агент по заданию сам собирает схему через типизированный API (skills).
- **Возможный облачный SaaS (мультиюзер)** — если проект «взлетит»: тот же `core` на сервере. Готовность закладываем сейчас (принцип 7), но поставка по умолчанию — десктоп.

## Стек (решение)

- **TypeScript**, монорепо (pnpm workspaces).
- `packages/core` — чистый TS, **БЕЗ UI**, работает в Node и в браузере (домен, связность, рендер в SVG, генераторы, валидация).
- `apps/desktop` — web-UI (SVG-канвас) на Vite. Нативная **оболочка (Tauri ИЛИ Electron) выбирается на Фазе 0.5** — выбор обратим: UI не зависит от оболочки, нативные вызовы (диалоги файлов, обновление) — за тонким `Platform`-портом.
- `packages/agent` — БУДУЩЕЕ: skills/tools поверх `core` для LLM-генерации.

## Архитектурные принципы (НЕ нарушать)

1. **Headless-first / API-first.** Всё, что делает пользователь мышкой, выражается вызовом к `core`.
   `core` обязан работать без GUI (рендер в SVG headless). GUI и LLM-агент — два равноправных клиента ядра.
2. **Связность вычисляется, не хранится.** Храним геометрию (где выводы и провода);
   цепи/потенциалы (`Net`) считаем через union-find. Из них — автонумерация, таблица соединений.
   **Провод (`Wire`) ≠ линия (`Line`)**: только провода электрические; вставка символа на провод его разрезает.
3. **Три слоя домена раздельны:** Каталог (изделия производителей) ↔ Символы (УГО-графика) ↔ Схема (чертёж).
   Одно устройство (`Device`) = 1 строка спецификации, но N представлений: графические `SymbolInstance`
   (master/slave) + физический footprint в шкафу (`CabinetItem`). У `Device` может быть несколько кодов каталога.
4. **Открытые форматы.** Символы и парты — JSON с JSON Schema. Проект — zip-контейнер `.seeproj`.
5. **Undo/Redo с самого начала** (command-паттерн). Встроить потом нельзя.
6. **Тесты — это «глаза» агента.** Юнит-тесты на `core` (особенно движок связности), snapshot-тесты SVG-рендера.
7. **Ядро готово к мультиюзеру/облаку «бесплатно».** В `core`: никакого прямого I/O (хранилище — через инжектируемый порт `Storage`); никаких глобальных синглтонов (много проектов/сессий в одном процессе); стабильные `uuid` у всех сущностей (не индексы массивов); проект сериализуется детерминированно + версия схемы и миграции; **команды (undo/redo) сериализуемы** → основа realtime-совместной работы. Авторизация/пользователи/тенанты — ВНЕ ядра.

## Эталоны

- **See Electrical** (установлен у пользователя локально) — эталон UX/эстетики/воркфлоу. Изучаем по скриншотам, эмулируем поведение. НЕ копируем ассеты/иконки/код.
- **QElectroTech** (GPL, открытый код) — эталон логики реализации (коннекторы, нумерация). Читаем, переносим идеи. НЕ форкаем (C++/GPL).
- **ГОСТ (ЕСКД)** — источник самих УГО, штампа, нумерации цепей. См. `docs/ARCHITECTURE.md`.

## Документы

- `docs/ARCHITECTURE.md` — доменная модель, движок связности, форматы.
- `docs/ROADMAP.md` — фазы с критериями «готово».
- `docs/SPRINTS.md` — индекс спринтов; сами спринты — `docs/sprints/S{N}-{slug}.md`.
- `docs/REFERENCE-SEE-ELECTRICAL.md` — разбор UX See Electrical → наш дизайн.

## Скиллы спринтов

Процесс спринтов ведётся через скиллы (`.claude/skills/`), адаптированные под этот проект (solo-агент, монорепо TS, ветки от `main`):

- **`/sprint [N]`** — статус спринта (прогресс по чеклисту `- [x]`/`- [ ]`, ближайшие/выполненные задачи). Read-only.
- **`/sprint-create [slug]`** — создать спринт по `docs/sprints/SPRINT-TEMPLATE.md`: согласование (если зона доменной модели / связности / формата `.seeproj`), контекст, автонумерация `S{N}`, синк `docs/ROADMAP.md` + индекса `docs/SPRINTS.md`, ветка от `main`.
- **`/sprint-finalize [N]`** — закрыть: итоговый отчёт в спринт-документе, статус `✅ Готов`, синк ROADMAP/индекса, уроки в memory, коммит.

Конвенции: спринты — `docs/sprints/S{N}-{slug}.md`; статусы `📝 Запланирован` / `🚧 В работе` / `✅ Готов` / `📝 Опционально`; прогресс — только по `- [x]`/`- [ ]`. Инфраструктура mastersmeta (agent-bus, `/dispatch`, forgejo, BACKLOG, cross-agent) здесь НЕ используется.

## Открытые вопросы

- Лицензия: код (MIT/Apache-2.0?) vs контент-библиотеки (CC-BY?).
- Точная вёрстка отчётов (типы известны: перечень аппаратуры/клемм/кабелей/жил/PLC/документов; шаблоны `.tdw` + Crystal Reports) — нужен скрин конкретного «Zestawienie» для колонок и ГОСТ-оформления.
- Tauri vs Electron для десктоп-обёртки.
